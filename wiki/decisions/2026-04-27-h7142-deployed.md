# Decision: H7142 deployed; PI controller promoted from shadow to authoritative

**Date:** 2026-04-27
**Status:** Implemented and deployed.
**Implements:** [2026-04-26 Govee humidifier pivot](2026-04-26-govee-humidifier-pivot.md).

## Context

The 2026-04-26 pivot decision committed to replacing the Raydrop + Kasa EP10 stack with a Wi-Fi-native GoveeLife H7142, integrated via the Govee Public API v2, and promoting the existing shadow-mode PI controller to authoritative once the new actuator landed. The H7142 arrived 2026-04-27; this decision records what actually shipped.

## What changed

### Hardware

- **Deployed:** GoveeLife H7142 (6 L cool-mist ultrasonic), DHCP-reserved at `192.168.1.247`, MAC `14:38:60:74:F4:DD:B9:46`, account name `dirt-humidifier`.
- **Retired (same day):** Raydrop 4L + Kasa EP10 plug. Both physically unplugged; Kasa plug deprovisioned from the loop. Raydrop retained as a spare; Kasa EP10 may be reused for a future actuator.
- **H7140 backup:** ordered earlier the same day en route as the de-risking backup (3 L Lite, 8 levels). Will be returned or retained as a spare — TBD.

### Software

Five new code paths landed in one PR:

1. **`apps/shared/src/dirt_shared/services/govee.py`** — async client wrapping `httpx.AsyncClient`. Methods: `discover()` (GET), `get_state()` (POST), `set_power()` (POST), `set_manual_level()` (POST). `GoveeError` / `GoveeRateLimitError` mapping. Pure transport — no retries; loop is the retry boundary. 14 tests via `httpx.MockTransport`.
2. **`apps/hwd/src/dirt_hwd/services/humidifier_dispatch.py`** — pure-function quantizer mapping `u_pct ∈ [0, 100] → modeValue ∈ {1..9}` with hysteresis at level boundaries. 20 property tests.
3. **`apps/hwd/src/dirt_hwd/services/humidifier.py`** — rewritten loop. PI compute → quantize → diff against live device state → minimal API calls. Replaces the bang-bang state machine and Kasa-plug dependency entirely. New helpers: `LackWaterState` rising-edge tracker, `IneffectiveState` watchdog (rebrand of `StuckState`, semantics now keyed on commanded level rather than plug ON/OFF), `_plan_dispatch` diff function. 18 helper unit tests + 6 single-tick integration tests.
4. **`apps/shared/src/dirt_shared/config.py`** — `HumidifierConfig` dropped `kasa_*` and `vpd_deadband_kpa`; added `govee_api_key`, `govee_sku`, `govee_mac`, `mist_levels`, `level_hysteresis_pct`, `pi_*` gain fields, and renamed `stuck_*` → `ineffective_*`. PI gains pulled out of module-level constants in `humidifier.py` so they're tunable via `.env` after the step-test refit.
5. **Atlas migration `20260428003304_add_govee_sensor_source.sql`** — adds `govee` to the `sensor_source` Postgres enum so the new `humidifier_on` / `humidifier_mist_level` breadcrumbs can record `source=govee`.

### Promotion of the PI controller

The PI controller in `humidifier_pi.py` has been live in shadow mode since 2026-04-25. Promoted to authoritative in this PR — same module, same gains (`Kc=8.0`, `Ki=0.01`), same guards (failsafe-stale-sensor / outside-lights-window / RH-ceiling). Just plumbed to a new dispatch boundary instead of being a side-channel diagnostic. The shadow stream stays alive as a "raw PI output before quantization" diagnostic.

### Observability changes

The `humidifier` log stream evolves:

| Event | Before | After |
|---|---|---|
| `state_change` | power on/off with reason/VPD/band | now also carries `level: int\|null`, `u_pct`, `bucket_width_pct` |
| `level_change` | — | NEW: level transitions while powered (from→to, u_pct, band) |
| `lack_water` / `lack_water_cleared` | — | NEW: rising/falling edge of `lackWaterEvent`; one Telegram per refill cycle |
| `device_offline` / `device_online` | — | NEW: Govee cloud reachability transitions |
| `suspected_ineffective` | was `suspected_stuck` (Raydrop red-LED-specific) | RENAMED + repurposed; semantics keyed on commanded mist level rather than Kasa plug state |
| `rate_limited` | — | NEW: HTTP/code 429 from Govee — quiet log; next tick retries naturally |
| `skip_offline` | — | NEW: per-tick when device is offline; PI ran but no actuation |

The `humidifier_shadow` stream stays unchanged — its analyzer + replay harness (`debug/humidifier-shadow/analyze.py`) keeps working without modification.

## Live discovery findings (vs. expectations)

The reference pack written 2026-04-26 had several speculative claims about the H7142. Live discovery on 2026-04-27 corrected them:

| Claim (pre-deploy) | Live (2026-04-27) | Action |
|---|---|---|
| `POST /user/devices` with empty body | **`GET /user/devices`** (POST returns 400) | Reference pack updated; client uses GET |
| Response field name is `msg` | `message` on GET, `msg` on POST endpoints | Client tolerates both |
| Response shape is `{data: {devices: [...]}}` | `{data: [...]}` directly on GET; client tolerates both | Client tolerates both |
| H7142 likely exposes UVC + aroma + nightlight capabilities | **None of these are exposed.** Capability list is exactly: `powerSwitch`, `workMode`, `humidity`, `lackWaterEvent` | Reference pack updated; loop binds only to the four |
| Auto-mode `humidity` setpoint range is 40–80% | Standalone `humidity` capability range is **40–70%**; Auto-mode `modeValue` (used inside `workMode`) range is 40–80%. The two are different surfaces. | Reference pack clarified; we don't use Auto, so no behavior change |
| Manual-mode `modeValue` range is 1..9 | **Confirmed 1..9** (matches marketing). `workMode` enum is `1=Manual, 2=Custom, 3=Auto`. | No change |

Probe scripts at `debug/govee_probe.py` and `debug/govee_power_cycle.py` are re-runnable for any future H7142 firmware updates.

## Rate-limit budget

Per-account: 10K/day. Per-device: ~10/min.

Steady-state per-tick traffic:
- 1× state read (always)
- 0–2× control calls (only when level needs to change, or boot tick)

At 30 s ticks → 2,880 ticks/day → 2,880 state reads/day baseline, plus control calls only on transitions. Even if every single tick required a control call (worst case), we'd hit ~5,760/day — still well under the daily ceiling. Per-minute on a single device peaks at 6 calls/min in the boot-tick + state-read combo, well under the 10/min ceiling.

## Migration plan vs. what actually happened

Per the proposal: land in two PRs (client + dispatch first, then loop rewrite + cutover). **Actual:** all five changes landed in one PR — the wiring was tightly coupled enough that splitting added more friction than it removed. Cutover via `systemctl --user restart dirt-hwd`. Raydrop + Kasa EP10 already physically unplugged at the user's end, so no parallel run.

The week-1 calibration plan stands: graduated step test for FOPDT refit, observe `humidifier_shadow` and `humidifier` streams, tune PI gains in `.env`, restart.

## What didn't change

Listed for clarity:

- **VPD-targeting decision** (2026-04-18). RH ceiling stays as an envelope guard, not a setpoint.
- **Lights window logic.** Still `lights_off_prep_minutes=5`. Still symmetric pre-lights-off and pre-lights-on.
- **Stage band table.** Veg/early-flower/late-flower VPD bands and RH ceilings unchanged.
- **`humidifier_pi.py`.** Module unchanged. Sample tests unchanged. The PI controller was always portable across actuators — proven now in production.
- **Daily-report pipeline.** Doesn't read humidifier-on/level; nothing to update.
- **`device_watchdog`, `metric_freshness` services.** Don't track the humidifier. No change.

## Acceptance signals (what we'll watch)

- 24h with VPD inside the veg band, no `suspected_ineffective`, no `error` events.
- `lackWaterEvent` rising-edge alert fires correctly on the next refill cycle.
- `level_change` cadence < 5/hour at steady state (otherwise tighten hysteresis).
- Steady-state ticks emit only the state read, no control calls (otherwise the diff is wrong).

## Rollback

The Kasa EP10 plug + Raydrop are intact. Roll back by:

1. Revert this PR (loop file + config + Govee client)
2. Plug Raydrop into Kasa EP10
3. `systemctl --user restart dirt-hwd`
4. The Atlas migration adding `govee` to `sensor_source` enum can stay — adding an enum value is forward-compatible with old code paths that don't use it.

Estimated rollback time: ~10 minutes. Unlikely to be needed — the H71xx capability set was verified live before the loop wiring, and the dispatch logic is property-tested.
