# Decision: GPIO3 + ESP-IDF ADC Driver on ESP32-C3 Plant Nodes

**Date:** 2026-04-14
**Status:** Accepted

## Context

The distributed sensor architecture decision (2026-04-12) specified GPIO0 for the per-plant soil moisture reading. During the first plant-a bring-up on 2026-04-14, we discovered two unrelated ESP32-C3 issues — one about pin choice, one about which ADC API to call.

## Decision

Two linked technical choices for all ESP32-C3 plant nodes:

1. **Use GPIO3 (ADC1_CH3) for the soil moisture analog input, not GPIO4.**
2. **Read the ADC via the ESP-IDF native `adc1_get_raw()` API, not Arduino's `analogRead()`.**

## Why GPIO3, not GPIO4

GPIO4–GPIO7 on the ESP32-C3 are JTAG pins (MTMS/MTDI/MTCK/MTDO). When WiFi or BT is initialized, the JTAG peripheral can be activated and drives these pins — any ADC reading on them collapses to ~0 regardless of what's connected.

Empirical confirmation 2026-04-14:
- Bench firmware, no WiFi, sensor on GPIO4: raw 2750 in air, 620 in water (expected).
- Production firmware with WiFi, same hardware: raw collapsed to 2–4.
- Moved sensor to GPIO3, same production firmware: readings returned to normal.

GPIO3 (ADC1_CH3) is the nearest safe neighbor — adjacent on the header, clean ADC1 channel, no JTAG conflict, no strapping-pin duties. GPIO0 and GPIO1 are also safe but further from the other pin header positions.

## Why `adc1_get_raw()`, not `analogRead()`

Multiple documented reports in `arduino-esp32` (issues [#102](https://github.com/espressif/arduino-esp32/issues/102), [#5188](https://github.com/espressif/arduino-esp32/issues/5188), [#5502](https://github.com/espressif/arduino-esp32/issues/5502), and [esp-idf-hal #468](https://github.com/esp-rs/esp-idf-hal/issues/468)) show ADC1 + WiFi instability when using `analogRead()` on the C3 — readings can stick at 0 or 4095, or return wildly wrong values, specifically when WiFi is active. The ESP-IDF driver (`adc1_config_width`, `adc1_config_channel_atten`, `adc1_get_raw`) bypasses the issue.

```cpp
#include <driver/adc.h>

// Setup:
adc1_config_width(ADC_WIDTH_BIT_12);
adc1_config_channel_atten(ADC1_CHANNEL_3, ADC_ATTEN_DB_11);

// Read:
int raw = adc1_get_raw(ADC1_CHANNEL_3);
```

Maps GPIO3 → ADC1_CHANNEL_3. Same 12-bit resolution, same ~0–3.3V input range via 11dB attenuation. Slightly more verbose than `analogRead(3)`; worth it for reliable readings.

## Alternatives Considered

| Option | Pros | Cons |
|--------|------|------|
| **GPIO3 + IDF driver** (chosen) | Reliable with WiFi, minimal footprint change | Slightly more verbose setup |
| GPIO4 + `analogRead()` (original plan) | Simplest | Broken with WiFi — JTAG + ADC stack-up |
| GPIO5 (ADC2) | Different pin | ADC2 is not usable with WiFi on C3 (documented by Espressif) |
| Toggle WiFi off before each read | No code changes | Breaks OTA responsiveness, adds latency, still relies on analogRead |

## Consequences

- All ESP32-C3 plant nodes wire the sensor's AOUT to silkscreen pin "3" (chip GPIO3). Board silkscreen "4" is physically nearby — **easy to mis-plug, verify after wiring**.
- Firmware includes `<driver/adc.h>` and uses `adc1_get_raw()` everywhere. The readings service doesn't care which API produced the raw value; this is internal to the firmware.
- The distributed sensor architecture epic documents GPIO3 as the standard; ADR 004 annotated accordingly.
